# AGENT.md 편집 규칙

이 문서는 프로젝트 루트 `AGENT.md`를 갱신할 때 Codex가 따라야 할 규칙이다.
사용자가 "AGENT.md에 반영해줘" 등으로 편집을 지시했을 때, **반드시 이 파일을 먼저 Read한 뒤 시작하라.**

---

## 1. AGENT.md의 6개 표준 헤더

AGENT.md는 다음 6개 헤더만 사용한다. **새 헤더를 만들지 마라.**

| 헤더 | 용도 | 예시 형식 |
|---|---|---|
| `## 기술 스택` | 핵심 프레임워크/언어만 5줄 이내 | `- Next.js 15` |
| `## 아키텍처 규칙` | 코드 보면 모를 강제 행동 (디렉토리 격리, 호출 경로 등) | `- CRITICAL: API 로직은 app/api/에서만. 이유: 보안.` |
| `## 개발 프로세스` | TDD, 커밋 컨벤션, 코드 리뷰 등 작업 방식 | `- CRITICAL: TDD. 새 기능은 테스트 먼저.` |
| `## 명령어` | 빌드/테스트/린트. 안 적으면 Codex가 매니페스트 추측하며 토큰 낭비 | `npm run dev` |
| `## 함정` | Codex가 반복해서 빠지는 실수 + 그 이유 | `- vitest만. jest 금지. 이유: 미설치.` |
| `## 도메인 메모` | 비즈니스 룰, 외부 시스템 계약 | `- 결제는 항상 멱등키 포함.` |

---

## 2. 추가할 때

- **사용자가 명시적으로 편집을 요청한 경우에만** 추가한다. 자체 판단으로 추가 금지.
- 위 6개 헤더 중 가장 맞는 곳에 넣는다.
- 각 항목은 **"X 하지 마라. 이유: Y"** 또는 **"X 해라. 이유: Y"** 형식. 추상적 서술 금지.
- **이유 없는 규칙은 추가하지 마라.** 이유가 빠지면 향후 판단 불가능해진다.
- 같은 함정이 이미 있는지 먼저 확인. 중복 추가 금지.

### 좋은 예
```
- 테스트는 vitest만. jest 시도 금지. 이유: 환경에 jest 미설치.
- DB 마이그레이션 자동 생성 금지. 무조건 수동 SQL.
  이유: 2025-11 ORM 자동 생성이 prod 인덱스를 잘못 만든 사고.
```

### 나쁜 예
```
- 테스트를 잘 작성하세요.              <!-- 추상적 -->
- jest 쓰지 마라.                      <!-- 이유 없음 -->
- src/components/에 컴포넌트 파일이 있습니다.   <!-- 코드 보면 앎, 행동 지침 아님 -->
```

---

## 3. 추가하지 마라 (안티패턴)

- **사용 라이브러리 전체 목록** — `package.json` 보면 안다. `## 기술 스택`엔 핵심만 5줄 이내.
- **폴더 구조 트리 / 파일 위치 나열** — 코드 보면 안다.
- **"잘 작성해라", "조심해라"** 같은 추상적 표현 — 행동을 바꾸지 못한다.
- **한 번만 쓸 일회성 노트** — 작업 메모는 별도 파일로.
- **일반 코딩 원칙** (Think Before Coding, Simplicity First 등) — 개인 글로벌 설정에 둘 것. 프로젝트 AGENT.md엔 프로젝트 특이 사항만.

---

## 4. 정리할 때

- AGENT.md가 **100줄을 초과**하면 사용자에게 분할을 제안한다:
  > "이 파일이 N줄입니다. `coding-style.md` 등으로 분리하고 여기엔 참조만 남길까요?"
- 효력이 사라진 함정(라이브러리/도구 변경으로 무의미해진 항목) 발견 시 사용자에게 정리 제안.
- 중복·충돌 규칙 발견 시 즉시 사용자에게 알린다.

---

## 5. 절대 하지 마라

- **사용자 승인 없이 기존 규칙 삭제 금지.**
- **사용자 승인 없이 헤더 구조 변경 금지.**
- "참고로 이것도 좋을 것 같습니다" 식 임의 추가 금지.
- **AGENT.md 상단의 이 파일(`.codex/AGENT-EDITING-RULES.md`) 참조 줄을 삭제하지 마라.** 향후 갱신 품질의 근거다.

---

## 6. 초기 셋업 (`AGENT.md 셋업해줘` 요청 시)

1. **제목**: `{프로젝트명}`을 실제 프로젝트명으로 교체.
2. **기술 스택**: 실제 프레임워크/언어/스타일링으로 placeholder 교체. **5줄 이내 유지.** 라이브러리 전체 나열 금지.
3. **아키텍처 규칙**: placeholder는 그대로 둔다. **Codex가 임의로 채우지 마라.** 사용자가 실제 규칙을 알려줄 때까지 대기.
4. **개발 프로세스**: TDD/commits 항목이 이 프로젝트에 적용되지 않으면 사용자에게 확인 후 제거.
5. **명령어**: `package.json` / `pyproject.toml` / `Cargo.toml` / `pom.xml` / `Makefile` 등 빌드 매니페스트를 읽어 실제 명령어로 교체. 없는 명령은 삭제.
6. **함정, 도메인 메모**: `(아직 없음)` 그대로 둔다. 작업 중 실수가 누적되면 사용자 지시로 채워진다.

---

## 7. 배경 (왜 이런 규칙인가)

이 규칙들은 안드레 카파시(Andrej Karpathy)의 65줄 LLM 가이드 원칙과 Anthropic 공식 작성 가이드를 종합한 것이다 (Claude·Codex 등 LLM 에이전트 전반에 동일하게 적용된다).

핵심 철학:
- AGENT.md는 **README가 아니라 가드레일**이다. 사람이 아니라 AI를 위한 문서.
- 처음부터 완벽하게 쓰려 하지 말고, **실수를 통해 키워라.**
- 모든 줄이 토큰이다. `execute.py`가 매 단계 AGENT.md를 통째로 동봉하므로 **100줄 이하 유지**가 비용에 직결된다.
- "X 하지 마라. 이유: Y" 형식이 추상적 가이드라인보다 항상 낫다.
